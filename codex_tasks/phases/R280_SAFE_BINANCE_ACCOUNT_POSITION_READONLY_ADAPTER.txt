Why this phase exists:
  R279 added Binance autonomous readiness binding.
  R279 safely verified public/read-only exchange facts:
    - BTCUSDT exchange filters
    - mark price
    - min notional
    - min quantity
    - step size
    - 80 USDT cap clears exchange minimum
  R279 intentionally blocked account/position inspection as:
    account_position_readiness_status="NOT_IMPLEMENTED_SAFELY"

  R280 must implement that missing piece safely:
    - read account/wallet balance
    - read BTCUSDT position risk / open position status
    - verify no conflicting live position
    - verify wallet can support tiny live
    - optionally read leverage/margin mode if safely exposed by existing Binance read-only surfaces
    - never place, test, modify, cancel, transfer, or withdraw

Hard goal:
  Add a safe private read-only Binance account/position adapter that can be used by R279/R280 readiness without creating any trading capability.

Non-negotiable safety:
  - Do NOT place real orders.
  - Do NOT call Binance /fapi/v1/order.
  - Do NOT call Binance test order endpoint.
  - Do NOT cancel orders.
  - Do NOT change leverage.
  - Do NOT change margin mode.
  - Do NOT transfer.
  - Do NOT withdraw.
  - Do NOT create signed trading requests.
  - Do NOT expose API keys or secrets.
  - Do NOT print raw secret values.
  - Do NOT mutate env files.
  - Do NOT enable real live execution.
  - Do NOT disable global kill switch.
  - Do NOT make final command available.
  - Do NOT arm live.
  - Do NOT treat read-only success as permission to execute.
  - If any endpoint or adapter uncertainty exists, block readiness.

Allowed endpoint class:
  Private read-only Binance endpoints only, behind explicit confirmation.

  Allowed examples, if implemented safely:
    - futures account balance/account read-only endpoint
    - futures position risk read-only endpoint
    - position mode read-only endpoint if needed and safe

  Forbidden examples:
    - /fapi/v1/order
    - /fapi/v1/batchOrders
    - /fapi/v1/leverage
    - /fapi/v1/marginType
    - cancel endpoints
    - transfer endpoints
    - withdraw endpoints

Implementation scope:
  1. Safe signed read-only adapter:
     Add or extend a module, preferably:
       src/app/hammer_radar/operator/binance_account_position_readonly.py

     It must expose safe functions similar to:
       build_signed_readonly_query(...)
       fetch_futures_account_balance_readonly(...)
       fetch_futures_position_risk_readonly(...)
       build_account_position_readiness(...)

     Requirements:
       - signing may be used ONLY for read-only endpoints.
       - signed_readonly_request_created may be true.
       - signed_trading_request_created must remain false.
       - signed_order_request_created must remain false.
       - hmac_signature_created may be true only if signed_readonly_request_created=true and endpoint is allowlisted read-only.
       - never expose signature, query string containing secret-derived material, api key, or secret.
       - endpoint allowlist must be explicit.
       - any non-allowlisted endpoint must block before request.

  2. Endpoint allowlist:
     Add a clear allowlist, for example:
       /fapi/v2/balance
