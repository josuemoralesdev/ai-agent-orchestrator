  Duplicate risk level: MEDIUM

Why this phase exists:
  The operator reached the R262B/R263/R264 JIT launch path for BTCUSDT tiny-live, but R264 dry preview blocks with:
    risk_contract_config_invalid

  Current observed state:
    - R262B contract-fit says risk_contract_valid=true.
    - R263 final console arms the experimental 8m short lane.
    - R264 dry preview says risk_contract_valid=false and blocks submit.
    - /readiness returns NOT_READY because there is no fresh ELIGIBLE_TINY_LIVE BTCUSDT candidate.
    - /trade-ticket returns BLOCKED because no current BTCUSDT live-checklist candidate exists.
    - /strategy-promotion/status says 13m long and 44m long are promotion-ready future tiny-live candidates.
    - 8m is paper-only by promotion config, while R263 temporarily arms it as an experimental lane.
    - The protocol endpoint says max_position_usd=44, preferred_leverage=2, max_leverage=3.
    - Some newer risk-contract/JIT artifacts are using tiny_live_margin_usdt=44 and leverage=10, producing ~399 USDT notional.
    - This creates a semantic mismatch: “44 USDT tiny live” may mean margin budget in one validator and position/notional cap in another.

Hard goal:
  Make the next Tiny Live path proper tiny live:
    - reconcile validator mismatch,
    - expose the final console/UI clearly,
    - make the operator-facing state obvious,
    - keep all real order submission blocked until the final valid JIT packet says go,
    - produce a clean manual-only path for the next fresh eligible candidate.

Non-negotiable safety rules:
  - Do NOT place real orders.
  - Do NOT call Binance /fapi/v1/order.
  - Do NOT call Binance test order endpoint unless an existing test explicitly already does so safely. Prefer no test-order calls.
  - Do NOT enable live execution globally.
  - Do NOT disable any kill switch.
  - Do NOT print secrets.
  - Do NOT persist secrets.
  - Do NOT mutate external env files.
  - Do NOT create a live submit command unless every safety and risk-contract gate is valid.
  - If any uncertainty exists, remain BLOCKED.
  - Codex must not run a final live submit.
  - Codex must not ask for sudo or SSH key passphrase.
