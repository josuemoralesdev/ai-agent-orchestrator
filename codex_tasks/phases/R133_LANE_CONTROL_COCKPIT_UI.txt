PHASE:
R133 Lane Control Cockpit UI

BRANCH:
r133-lane-control-cockpit-ui

PHASE CLASSIFICATION:
Primary: WIRING / INTEGRATION
Secondary: EXTENSION OF EXISTING CAPABILITY, DIAGNOSTIC / AUDIT, DUPLICATE RISK
Duplicate risk: MEDIUM

GOAL:
Create a read-only operator cockpit UI for the autonomous lane architecture.

Design direction:
Option D:
Thread Launcher-style Control Tower + MasonShift sacred gate accents.

This UI must be aesthetic, compact, operational, and safe.
It must show the lane system clearly without exposing dangerous controls.

CORE UX PRINCIPLE:
The operator should understand the whole system in 5 seconds:

- Is the system safe?
- Which lanes are enabled?
- Which lanes are paper / armed dry run / tiny_live?
- Is there a fresh routed signal?
- Is autonomous paper proof present?
- Is tiny-live gate blocked or near-ready?
- What is the next safest action?

NON-NEGOTIABLES:
- Read-only UI only.
- Do not place real orders.
- Do not create Binance order payloads.
- Do not call Binance order endpoints.
- Do not send signed requests.
- Do not print secrets.
- Do not mutate env files.
- Do not mutate lane config.
- Do not enable global live execution.
- Do not bypass R106/global gates.
- Do not alter global live gates to make tiny_live easier.
- Do not create live action buttons.
- Do not create order buttons.
- Do not install/start services.
- Do not run sudo.
- Do not commit.
- Do not merge.
- Do not tag.

IMPORTANT:
R133 is a cockpit UI, not an execution interface.
Allowed UI actions:
- refresh
- copy command
- view blockers
- inspect lane details

Forbidden UI actions:
- place order
- enable live
- bypass gate
- send to Binance
- mutate env
- write lane config
