WHY THIS PHASE EXISTS:
Recent phases exposed an architecture smell.

We keep adding:
- new timeframes
- new signal origins
- new pattern families
- WMA/MA anchors
- betrayal candidates
- entry modes
- lane matrices
- source emitters
- direction split rules
- evidence requirements
- safety gates

But too much is being stitched directly into each phase.

That creates duplicate-risk and memory-risk:
- betrayal can be forgotten unless explicitly named
- 222m / 88m / 55m can be missed
- entry modes get assumed instead of declared
- source_identity gets lost
- direction split rules drift across modules
- each new phase reinvents its own target list

R217 showed the latest example:
- ready_rows=0
- partial_rows=9
- blocked_rows=5
- decomposition_status=AGGREGATE_DECOMPOSITION_REQUIRES_ENTRY_MODE
- missing fields are entry_mode and source_identity
- future rows need an explicit registry/manifest layer so emitters/resolvers know what is required

R218 must create a central paper-only strategy evidence registry / source identity manifest.

PURPOSE:
Create a registry/manifest layer that:
1. Defines known timeframes in one place.
2. Defines known entry modes in one place.
3. Defines known signal origins in one place.
4. Defines betrayal candidates in one place.
5. Defines direction rules in one place.
6. Defines source identity requirements in one place.
7. Defines evidence requirements by source family.
8. Defines safety status by capability family.

   - what timeframes are in scope?
   - what entry modes are valid?
   - what signal origins exist?
   - what betrayal candidates exist?
   - what fields are required for resolver-ready source rows?
