Duplicate risk: HIGH

GOAL:
Fix the signal-to-watcher gap discovered by R144.

Visible BTCUSDT signals can match watched lane symbol/timeframe/direction, but raw signal records often have:
entry_mode = null

R144 proved fresh_signal_router can derive:
ladder_close_50_618

and derive lane keys like:
BTCUSDT|13m|long|ladder_close_50_618
BTCUSDT|44m|long|ladder_close_50_618

But R142 watcher eligibility still treats raw entry_mode null as not exactly matching watched lane keys.

R145 must create a safe normalization bridge so BTCUSDT 13m/44m long signals with missing entry_mode can be normalized into the correct watched lane key before watcher/paper eligibility decisions.

This is NOT live execution.
This is NOT Binance.
This is NOT order placement.
This is a lane-candidate normalization bridge.

TARGET LANES:
- BTCUSDT|13m|long|ladder_close_50_618
- BTCUSDT|44m|long|ladder_close_50_618

CURRENT VERIFIED CONTEXT:
R143 recorded tiny-live lane unlock contract:
- execution_state: UNLOCKED_WAITING_FOR_CONDITIONS
- lanes: 13m long ladder_close_50_618 and 44m long ladder_close_50_618
- no order/Binance/payload/env/global mutation

R144 trace found:
- SIGNAL_MATCHES_WATCHED_LANE_BUT_ENTRY_MODE_MISSING: 10
- SIGNAL_TIMEFRAME_NOT_WATCHED: 84
- SIGNAL_DIRECTION_NOT_WATCHED: 6

Specific 44m long signal:
BTCUSDT|44m|long|2026-05-28T12:39:59.999000+00:00
- entry_mode: null
- fresh_router_derived_entry_mode: ladder_close_50_618
- derived_lane_key: BTCUSDT|44m|long|ladder_close_50_618
- gap: SIGNAL_MATCHES_WATCHED_LANE_BUT_ENTRY_MODE_MISSING
- route_status after delay: EXPIRED_SIGNAL
- paper_executor blocked because stale, but the lane derivation itself is valid

DESIRED RESULT:
When a fresh BTCUSDT 13m/44m long signal appears with entry_mode null:
- derive entry_mode = ladder_close_50_618 through the same logic used by fresh_signal_router
- normalize candidate lane_key
- allow R142/R129 paper proof path to evaluate it as the watched unlocked lane
- if fresh and otherwise eligible, paper proof capture can proceed
- if stale, it must remain blocked as stale
- no live execution

NORMALIZATION RULES:
1. Only normalize BTCUSDT signals/candidates.
2. Only normalize lanes included in R143 unlocked contract or explicit CLI lane keys.
3. Only normalize watched long lanes:
   - 13m long
   - 44m long
4. If entry_mode already exists:
   - preserve it.
5. If entry_mode is null and symbol/timeframe/direction match unlocked watched lane:
   - derive entry_mode = ladder_close_50_618
   - derive lane_key = symbol|timeframe|direction|ladder_close_50_618
   - mark derived_entry_mode = true
   - mark derivation_source = "R145_ENTRY_MODE_DERIVATION_BRIDGE"
6. If signal is stale:
   - normalization may occur, but eligibility must remain blocked by freshness.
7. If timeframe/direction not watched:
   - do not normalize into a watched lane.
8. Do not change raw log records in place.
9. Do not mutate signals.ndjson or historical ledgers.
10. Normalization is runtime/read-path only unless an explicit R145 bridge ledger record is requested.
