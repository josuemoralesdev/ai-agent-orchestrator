Implementation scope:
  1. New interpretation module:
     Add:
       src/app/hammer_radar/operator/tiny_live_leverage_margin_readiness.py

     It should build packet:
       event_type="TINY_LIVE_LEVERAGE_MARGIN_READINESS"
       created_by_phase="R283_LEVERAGE_MARGIN_READINESS_INTERPRETATION_FOR_ONE_SHOT_TINY_LIVE"
       status:
         LEVERAGE_MARGIN_READY
         LEVERAGE_MARGIN_BLOCKED
         LEVERAGE_MARGIN_NOT_CHECKED
         LEVERAGE_MARGIN_REQUIRES_MANUAL_OPERATOR_REVIEW

     Inputs should come from R282 account/position binding where possible:
       - configured_leverage=10
       - configured_margin_mode="isolated"
       - current_position_amt
       - current_position_notional
       - current_position_side
       - current_leverage from Binance read-only fields
       - current_margin_type/margin_mode from Binance read-only fields if available
       - leverage_matches_expectation
       - margin_mode_matches_expectation

  2. Classification logic:
     Safe classes:
       - EXACT_MATCH:
           current leverage == 10 and margin mode isolated.
           status LEVERAGE_MARGIN_READY.
       - ZERO_POSITION_METADATA_MISMATCH:
           current BTCUSDT position amount == 0 and notional == 0, but leverage/margin fields do not match or are missing.
           status should be LEVERAGE_MARGIN_REQUIRES_MANUAL_OPERATOR_REVIEW unless existing Binance behavior proves it harmless.
           final command remains unavailable.
       - NONZERO_POSITION_MISMATCH:
           any open BTCUSDT position and mismatch.
           status LEVERAGE_MARGIN_BLOCKED.
       - UNKNOWN_FIELDS:
           fields unavailable or unparsable.
           status LEVERAGE_MARGIN_BLOCKED or REQUIRES_MANUAL_OPERATOR_REVIEW, but not READY.
       - MUTATION_REQUIRED:
           if the only way to satisfy expected leverage/margin is mutation.
           status LEVERAGE_MARGIN_BLOCKED.
           Include manual_only_adjustment_required=true.
           Never perform mutation.

     Important:
       Do not mark READY just because there is zero position unless the code can prove order placement would use requested leverage/margin safely without mutation.
       Keep conservative.

  3. Output packet fields:
     - status
     - symbol BTCUSDT
