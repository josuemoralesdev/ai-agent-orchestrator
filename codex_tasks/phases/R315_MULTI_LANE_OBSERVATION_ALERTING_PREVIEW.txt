  It must preview operator/Telegram alerts only when something meaningful is degraded or unsafe.
  It must NOT send Telegram messages yet.

  No “everything is fine” alerts every minute.

  Operator attention is the scarce resource, because apparently humans only get one brain and keep filling it with notifications.

B. Alert rule engine:
  Use R314 health panel output as source.
  Alert conditions:
    - health_status == MULTI_LANE_OBSERVATION_HEALTH_BLOCKED
    - stale observation tick:
        last_tick_recent=false
        last_tick_age_seconds > max_age_seconds
    - timer not installed/enabled/active
    - service_last_exit_status not 0
    - any primary contract invalid
    - any primary observation status not OK
    - candidate freshness status critical, if such signal exists
    - final live safety violation:
        real_order_forbidden != true
        submit_allowed != false
        final_command_available != false
    - armed lane changed unexpectedly
    - safety flag violation:
        any required false flag true
        global_kill_switch != true
    - paper refresh critical failure
    - paper refresh degraded for tasks beyond accepted eth_paper_outcome-only degradation

Non-alert conditions:
    - health_status OK
    - current_candidate_seen=false
    - candidate_freshness_status=FRESH_TRIGGER_WAIT
    - paper_refresh_health_status=PAPER_REFRESH_DEGRADED_NON_CRITICAL with last_failed_tasks exactly or only eth_paper_outcome
    - normal timer tick
    - no candidate
    - secondary watch-only present
    - betrayal blocked preview-only

Alert severity:
  - INFO_PREVIEW_NO_SEND for no actionable alert
  - WARNING_PREVIEW_NO_SEND for degraded but safe
  - CRITICAL_PREVIEW_NO_SEND for blocked or safety violation
