GOAL:
Create autonomous paper lane execution for fresh routed signals using the R122/R123/R124 lane-control architecture.

WHY:

CORE CONCEPT:
A fresh routed candidate may produce a paper lane execution only if:

1. route_status is ROUTED_TO_LANE
2. lane mode is paper or armed_dry_run
3. signal is fresh under lane freshness_seconds
4. lane max_daily_trades is not exceeded
5. lane cooldown_after_loss_minutes is not active
6. paper/live separation remains intact
7. safety fields remain false

tiny_live mode:
- In R125, tiny_live lanes must NOT create real orders.
- tiny_live may create a PAPER_SHADOW_FOR_TINY_LIVE record only if safety allows.
- It must clearly say no real order was placed.

Default behavior:

- does not write paper execution record
- shows which candidates would be recorded

Supported args:
- --execute-paper
- --lane-key <lane_key>
- --all-lanes
- --confirm-paper-only "<exact phrase>"

Exact confirmation phrase:
I CONFIRM PAPER LANE EXECUTION ONLY; NO REAL ORDER; NO BINANCE CALL.

Behavior:
1. Without --execute-paper:
   - status: PAPER_LANE_PREVIEW
   - no ledger execution record written
   - shows routed fresh candidates and blockers
   - safety fields false

2. With --execute-paper but missing/wrong confirmation:
   - status: PAPER_LANE_REJECTED
   - no execution record written
   - rejection_reason: missing or invalid paper-only confirmation

3. With --execute-paper and exact confirmation:
   - status: PAPER_LANE_RECORDED or PAPER_LANE_PARTIAL
   - append paper execution records only for eligible routed candidates
   - never place orders
   - never create order payloads
   - never call Binance
   - never mutate env
   - return recorded paper_execution_ids
