# R270B — Strategy-Qualified Lane Risk Contract Selection for Tiny Live

  /home/josue/workspace/kernel/ai-agent-orchestrator-main

Phase classification:
  Primary: WIRING / SAFETY HARDENING
  Secondary: STRATEGY QUALIFICATION, RISK CONTRACT SELECTION, FINAL READINESS
  Duplicate risk level: LOW-MEDIUM

Why this phase exists:
  R270 correctly made manual unlock lane-aware and exposed signal origin / Betrayal status.
  It found the current proposed ticket lane:
    BTCUSDT|4m|long|ladder_close_50_618
  and classified it as:
    standard checklist
    betrayal_mode_involved=false
    betrayal_inverse_involved=false

  But R270 revealed a design issue:
    The explicit 80 USDT / 10x risk contract was scoped to a specific lane.
    The system should not hardcode only 4m long or 8m short.
    Operator policy is now:
      Allow any BTCUSDT lane/timeframe/strategy to be considered for tiny live if its tested win rate is above 55%, with sufficient sample size, explicit lane contract, and safety gates.

Hard goal:
  Replace single-lane hardcoding with a strategy-qualified lane selection policy:
    - Any lane/timeframe can be eligible if strategy evidence passes policy.
    - Default policy threshold: win_rate_pct >= 55.0
    - Minimum sample count must be enforced from existing promotion config or a safe default, preferably >= 30.
    - Exact lane risk contract must exist or be generated by explicit guarded apply path.
    - No live submit from Codex.
    - No cross-lane risk contract borrowing.

Operator policy:
  Tiny live lane eligibility may include all BTCUSDT timeframes and directions if:
    - win_rate_pct >= 55.0
    - sample_count >= configured minimum, or 30 if missing
    - strategy has valid entry_mode
    - risk contract is explicit for the exact lane
    - signal origin is standard or explicitly allowed
    - Betrayal/inverse is not involved unless separately accepted by explicit future gate
    - max notional remains 80 USDT
    - leverage remains 10x

Non-negotiable safety:
  - Do NOT place real orders.
  - Do NOT call Binance /fapi/v1/order.
  - Do NOT call Binance test order endpoint.
  - Do NOT enable live execution.
  - Do NOT disable kill switches.
  - Do NOT execute final submit.
  - Do NOT make final live command available in diagnostic mode.
  - Do NOT allow lane to borrow another lane’s risk contract.
  - Do NOT allow win-rate-only promotion without sample-count and safety gates.
  - Do NOT allow Betrayal/inverse first live unless future explicit acceptance exists.
  - If unsure, remain BLOCKED.

Implementation:
  1. Strategy-qualified lane policy:
     - Create/reuse a shared lane qualification helper.
     - A lane qualifies for tiny-live consideration if:
         win_rate_pct >= 55.0
         sample_count >= min_sample
         symbol == BTCUSDT
         entry_mode exists
         direction in {long, short}
         timeframe exists
     - Use configured min_sample from strategy promotion config if available.
     - Surface qualification reason and raw evidence.

  2. Exact lane risk contract:
     - For current proposed ticket lane, look for exact matching risk contract:
         symbol
         timeframe
         direction
         entry_mode
     - If missing but lane is strategy-qualified, create a guarded explicit contract record:
         tiny_live_contract_mode: explicit_notional_cap_with_leverage
         max_position_notional_usdt: 80.0
         max_notional_usdt: 80.0
         leverage: 10.0
         margin_budget_usdt: 8.0
         max_loss_usdt: 4.44 unless stricter existing policy applies
         live_execution_enabled: false
         live_authorized: false
     - Config write allowed only for the exact lane and only as risk contract, not live enablement.
     - Do not mutate lane_controls to arm live unless existing R263 flow explicitly does so later.

  3. No cross-lane borrowing:
     - 4m long cannot borrow 8m short contract.
     - 13m long cannot borrow 44m long contract.
     - Exact lane or BLOCKED.

  4. Trade ticket / final console / JIT:
     - Surface:
         current proposed ticket lane
         strategy qualification status
         win_rate_pct
         sample_count
         min_sample
         signal origin
         betrayal/inverse status
         exact risk contract status
         expected orders based on direction
     - Long expected orders:
         main BUY MARKET
         stop SELL STOP_MARKET REDUCE_ONLY
         take_profit SELL TAKE_PROFIT_MARKET REDUCE_ONLY
     - Short expected orders:
         main SELL MARKET
         stop BUY STOP_MARKET REDUCE_ONLY
         take_profit BUY TAKE_PROFIT_MARKET REDUCE_ONLY
     - Final command remains unavailable in diagnostic mode.

  5. Betrayal visibility:
     - Preserve:
         signal_origin_family
         betrayal_mode_involved
         betrayal_inverse_involved
     - If betrayal or inverse true:
         block with betrayal_first_tiny_live_not_explicitly_accepted.
     - If origin unknown:
         block or require manual_origin_review unless existing policy explicitly allows unknown.
       Prefer blocking.

  6. Tests:
     Add/update tests proving:
       - lane with win_rate >=55 and sample_count >= min_sample qualifies.
       - lane below 55 blocks.
       - lane below sample minimum blocks.
       - exact risk contract required.
       - missing exact contract can be created only by guarded policy for qualified standard lane.
       - no cross-lane contract borrowing.
       - 4m long contract validates if evidence qualifies.
       - long expected orders are BUY main / SELL exits.
       - short expected orders are SELL main / BUY exits.
       - Betrayal/inverse blocks unless explicitly accepted.
       - diagnostic mode never exposes executable submit command.
       - order_placed=false.
       - submit_attempted=false.
       - binance_order_endpoint_called=false.
       - secrets_shown=false.
