Why this phase exists:
  R287 proves the autonomous trigger loop semantics.
  R288 must make the loop runnable repeatedly, as a scheduler/service style dry-run loop, without requiring manual CLI pokes.
  This prepares the machine to watch overnight/daytime and record trigger states.

Hard goal:
  Add a safe scheduler loop wrapper for:
    tiny-live-autonomous-trigger-loop

  It should run repeatedly in dry-run mode, record packets, expose status, and remain safe:
    - no live order
    - no order endpoint
    - no executable payload
    - no final command
    - no per-signal operator approval

Non-negotiable safety:
  - Do NOT place orders.
  - Do NOT call Binance /fapi/v1/order.
  - Do NOT call Binance test order endpoint.
  - Do NOT call Binance leverage mutation endpoint.
  - Do NOT call Binance margin type mutation endpoint.
  - Do NOT call any mutation endpoint.
  - Do NOT enable live execution.
  - Do NOT disable global kill switch.
  - Do NOT write live config.
  - Do NOT mutate risk contracts.
  - Do NOT mutate env files.
  - Do NOT make final live submit command available.
  - Do NOT create executable order payloads.
  - Do NOT print API keys/secrets/signatures/signed URLs.
  - No per-signal approval gates.
  - Scheduler is dry-run visibility and audit only.

Allowed:
  - Build scheduler module.
  - Build CLI command to run one iteration.
  - Build CLI command to run bounded loop with limited iterations.
  - Build API status endpoint.
  - Build docs/tests.
  - Read-only Binance checks through safe loader/confirmation if explicitly requested by CLI.
  - Record dry-run loop packets.
  - Produce Telegram-compatible visibility payload prepared-not-sent by default.
  - Optionally send Telegram only if explicit send flag already supported and safe; default false.
