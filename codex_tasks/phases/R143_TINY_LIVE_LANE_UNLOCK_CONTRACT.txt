PHASE:
R143 Tiny-Live Lane Unlock Contract

BRANCH:
r143-tiny-live-lane-unlock-contract

PHASE CLASSIFICATION:
Primary: WIRING / INTEGRATION
Secondary: EXTENSION OF EXISTING CAPABILITY, DIAGNOSTIC / AUDIT
Duplicate risk: HIGH

GOAL:
Implement an operator-authorized tiny-live lane unlock contract that allows selected lanes to be marked as unlocked/armed-for-conditions without requiring a fresh candidate at unlock time.

This phase must separate:
1. Lane unlock / operator intent
2. Live execution conditions
3. Actual order placement

R143 must allow the operator to unlock specific lanes today, while ensuring no live order can happen unless future execution gates, fresh candidate checks, risk contracts, protective policy, environment flags, and kill-switch conditions all pass.

The desired state is:
UNLOCKED_WAITING_FOR_CONDITIONS

Not:
LIVE_ORDER_READY
Not:
ORDER_PLACED
Not:
BINANCE_CALLED

TARGET LANES:
- BTCUSDT|13m|long|ladder_close_50_618
- BTCUSDT|44m|long|ladder_close_50_618

WHY:
The system currently waits for fresh candidate before clearing too many operator gates. That makes the operator chase signals.
The correct design is:
- operator unlocks approved lane/timeframe/strategy in advance
- system waits for fresh candidate
- system enters only if all autonomous conditions are met
- no manual approval per signal

CURRENT CONTEXT:
R142 watcher works but times out because no watcher-eligible candidate appears.
Visible signals exist in signals.ndjson, but entry_mode is often null and not necessarily watcher-eligible.
The core architecture target is fully automated trading after operator arming:
- entry automated
- position size automated
- take profit/exit automated
- risk handling automated
- operator controls lane/timeframe/strategy ON/OFF
- operator has global kill switch
- operator does not approve individual signals after they appear
