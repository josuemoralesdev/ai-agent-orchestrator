Why this phase exists:
  Tiny Live final gate is now in the correct safe waiting state:
    FINAL_TINY_LIVE_AUTHORIZATION_WAITING_FOR_REAL_CANDIDATE
    timer_active=true
    timer_health_status=TIMER_HEALTH_ACTIVE
    blockers=[]
    Telegram polling confirmed end-to-end
    no duplicate dangerous processes found
  But the system has two real operational problems:
    1. hammer-paper-refresh.service repeatedly exits after completed=5 failed=1 and stopping=max_errors, but exits with status=0/SUCCESS, so systemd does not restart it.
    2. The current live wait is extremely narrow:
         BTCUSDT|44m|long|ladder_close_50_618 only
       This may be safe but too sparse. We need a paper-only Strategy Lab preview to evaluate broader candidates without changing live execution.

Purpose:
  Harden the paper refresh watcher so one non-critical paper-side failure cannot silently kill the refresh loop.
  Expose the exact failing task by name, reason, exception class, and recoverability.
  Add a Strategy Lab preview surface that evaluates live-qualified, near-miss, and betrayal/inverse candidates in paper-only mode.
  Keep Tiny Live as north star but do not mutate live flags or place orders.

A. Paper refresh durability requirements:
  Locate the current failure accounting in paper_refresh_scheduler.
  Make each task result include:
    - task name
    - status: completed / failed / skipped
    - recoverability: critical / non_critical / unknown
    - exception_class if failed
    - error_message redacted/sanitized
    - duration_ms
    - live_execution_enabled
    - order_placed
  Preserve existing successful output shape as much as possible.
  If one non-critical task fails, the watch loop must continue and write a degraded run record.
  If a critical task fails repeatedly, the service may stop, but it must exit nonzero so systemd Restart=on-failure can work.
  Add a clear summary:
    paper_refresh_health_status:
      PAPER_REFRESH_HEALTHY
      PAPER_REFRESH_DEGRADED_NON_CRITICAL
      PAPER_REFRESH_CRITICAL_FAILURE
      PAPER_REFRESH_STOPPING_MAX_ERRORS
  Add max_errors handling that distinguishes:
    - critical failures that should stop
    - non-critical failures that should keep the loop alive
  Never let order_placed become true from this scheduler.
